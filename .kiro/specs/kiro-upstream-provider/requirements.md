# Requirements Document

## Introduction

This document specifies the requirements for adding a Kiro upstream provider to the codex2claudecode project. The Kiro upstream provider enables the Claude inbound provider to route requests through the Kiro API (Amazon Q Developer) as an alternative to the existing Codex upstream provider. Phase 1 supports only the Claude inbound provider; the OpenAI inbound provider is not supported with Kiro upstream because Kiro never returns passthrough responses and the OpenAI inbound provider only handles `Canonical_PassthroughResponse` and `Canonical_ErrorResponse`. The provider reads authentication credentials from the local Kiro IDE SSO cache, refreshes tokens when expired, converts canonical requests to the Kiro `generateAssistantResponse` wire format, parses the AWS event-stream response back into canonical types, and lists available models via the `ListAvailableModels` API. Phase 1 covers single-account authentication (supporting both Kiro Desktop Auth and AWS SSO OIDC refresh flows, selected at init based on credential detection), payload conversion, streaming, and model listing. Multi-account, SQLite sources, VPN proxy, web_search MCP emulation, and `server_tool` type content block handling are excluded from phase 1.

## Glossary

- **Kiro_Auth_Manager**: The module responsible for reading, validating, and refreshing Kiro authentication tokens from the local SSO cache file. Supports two refresh methods: Kiro Desktop Auth and AWS SSO OIDC, selected at initialization based on credential detection.
- **Kiro_Client**: The HTTP client module that sends requests to the Kiro API with retry logic, token refresh on 403, and exponential backoff on 429/5xx.
- **Kiro_Payload_Converter**: The module that converts a Canonical_Request into the Kiro `generateAssistantResponse` wire format payload.
- **Kiro_Stream_Parser**: The module that parses the Kiro AWS event-stream binary response into canonical event and response types.
- **Kiro_Model_Resolver**: The module that normalizes model names and resolves model identifiers for the Kiro API.
- **Kiro_Upstream_Provider**: The top-level upstream provider class implementing the `Upstream_Provider` interface, orchestrating auth, client, payload conversion, and response parsing.
- **Auth_Token_File**: The JSON file at `~/.aws/sso/cache/kiro-auth-token.json` containing required fields `accessToken`, `refreshToken`, `expiresAt`, `region`, and optional fields `clientIdHash`, `clientId`, `clientSecret`, and `profileArn`. The `clientId`/`clientSecret` fields, when present directly in the Auth_Token_File, provide OIDC credentials without requiring a companion Device_Registration_File. When both direct credentials and a Device_Registration_File (via `clientIdHash`) are available, the direct Auth_Token_File credentials take priority.
- **Device_Registration_File**: The JSON file at `~/.aws/sso/cache/{clientIdHash}.json` containing `clientId` and `clientSecret` needed for SSO OIDC token refresh.
- **Canonical_Request**: The provider-agnostic request type defined in `src/core/canonical.ts`.
- **Canonical_Response**: The provider-agnostic non-streaming response type defined in `src/core/canonical.ts`.
- **Canonical_StreamResponse**: The provider-agnostic streaming response type defined in `src/core/canonical.ts`.
- **Canonical_Event**: The provider-agnostic stream event type defined in `src/core/canonical.ts`.
- **Canonical_InputMessage**: The provider-agnostic input message type defined in `src/core/canonical.ts`, containing `role` (`user`, `assistant`, or `tool`) and `content` (an array of `JsonObject` items). Content items use provider-shaped types such as `function_call` (for tool calls from assistant) and `function_call_output` (for tool results from tool role messages).
- **AWS_Event_Stream**: The binary framing format used by the Kiro API for streaming responses, containing JSON event payloads delimited by `:message-type` markers.
- **Kiro_API_Host**: The endpoint `https://q.{region}.amazonaws.com` used for `generateAssistantResponse` and `ListAvailableModels`.
- **Kiro_Refresh_Endpoint**: The endpoint `https://prod.{region}.auth.desktop.kiro.dev/refreshToken` used for Kiro Desktop Auth token refresh.
- **SSO_OIDC_Endpoint**: The endpoint `https://oidc.{region}.amazonaws.com/token` used for AWS SSO OIDC token refresh when `clientId` and `clientSecret` are available from the Device_Registration_File.
- **ConversationState**: The Kiro API payload structure containing `conversationId`, `currentMessage`, `chatTriggerType`, and optionally `history`.
- **Bootstrap**: The composition root at `src/app/bootstrap.ts` that wires concrete providers based on configuration.
- **Claude_Inbound_Provider**: The inbound provider at `src/inbound/claude/` that translates Claude Messages API format to/from canonical types. Phase 1 Kiro upstream only supports this inbound provider.
- **OpenAI_Inbound_Provider**: The inbound provider at `src/inbound/openai/` that handles OpenAI Responses/Chat Completions API format. Not supported with Kiro upstream in phase 1 because it only handles `Canonical_PassthroughResponse` and `Canonical_ErrorResponse`.

## Requirements

### Requirement 1: Auth Token File Reading

**User Story:** As a developer, I want the Kiro upstream provider to read authentication credentials from the local Kiro IDE SSO cache, so that I can use my existing Kiro IDE login without manual configuration.

#### Acceptance Criteria

1. WHEN the Kiro_Auth_Manager is initialized with a file path, THE Kiro_Auth_Manager SHALL read and parse the Auth_Token_File at the specified path as JSON.
2. WHEN no explicit file path is provided, THE Kiro_Auth_Manager SHALL default to reading `~/.aws/sso/cache/kiro-auth-token.json`.
3. WHEN the Auth_Token_File contains `accessToken`, `refreshToken`, `expiresAt`, and `region` fields, THE Kiro_Auth_Manager SHALL store each field for subsequent use. WHEN the Auth_Token_File also contains optional `clientId` and `clientSecret` fields, THE Kiro_Auth_Manager SHALL store them as direct OIDC credentials for potential use in SSO OIDC token refresh (see R3 criteria 1-2 for auth type detection priority). WHEN the Auth_Token_File also contains an optional `clientIdHash` field, THE Kiro_Auth_Manager SHALL store it and proceed to read the Device_Registration_File (per criterion 4). WHEN the Auth_Token_File also contains an optional `profileArn` field, THE Kiro_Auth_Manager SHALL store it for subsequent use in payload conversion (R6 criterion 1a) and model listing (R12 criterion 1). The `clientIdHash`, `clientId`, `clientSecret`, and `profileArn` fields are optional because Desktop Auth token files may not contain them; the four core fields (`accessToken`, `refreshToken`, `expiresAt`, `region`) are the minimum required set.
4. WHEN the Auth_Token_File contains a `clientIdHash` field, THE Kiro_Auth_Manager SHALL read the Device_Registration_File at `~/.aws/sso/cache/{clientIdHash}.json` and extract `clientId` and `clientSecret` for use in SSO OIDC token refresh. WHEN the Auth_Token_File does not contain a `clientIdHash` field, THE Kiro_Auth_Manager SHALL skip the Device_Registration_File read and proceed without `clientId` or `clientSecret`, allowing auth type detection (R3 criteria 1-2) to select Kiro Desktop Auth.
5. IF the Auth_Token_File does not exist at the specified path, THEN THE Kiro_Auth_Manager SHALL throw an error with a message identifying the missing file path.
6. IF the Auth_Token_File contains invalid JSON, THEN THE Kiro_Auth_Manager SHALL throw an error with a message describing the parse failure.
7. IF the Device_Registration_File does not exist, THEN THE Kiro_Auth_Manager SHALL log a warning indicating that the device registration file for the given `clientIdHash` is missing and that SSO OIDC refresh will not be available, and proceed without storing `clientId` or `clientSecret`. This allows the auth type detection (R3 criteria 1-2) to fall back to Kiro Desktop Auth when the companion file is missing.

### Requirement 2: Token Expiration Check

**User Story:** As a developer, I want the provider to detect when tokens are expired or expiring soon, so that requests are not sent with stale credentials.

#### Acceptance Criteria

1. THE Kiro_Auth_Manager SHALL parse the `expiresAt` field as an ISO 8601 datetime string.
2. WHEN the current time is within 600 seconds of the `expiresAt` timestamp, THE Kiro_Auth_Manager SHALL report the token as expiring soon.
3. WHEN the current time is past the `expiresAt` timestamp, THE Kiro_Auth_Manager SHALL report the token as expired.
4. IF the `expiresAt` field is missing or unparseable, THEN THE Kiro_Auth_Manager SHALL treat the token as expired.


### Requirement 3: Token Refresh

**User Story:** As a developer, I want the provider to automatically refresh expired tokens using the appropriate auth flow detected at initialization, so that long-running sessions continue without manual intervention.

#### Acceptance Criteria

1. WHEN the Kiro_Auth_Manager is initialized and `clientId` and `clientSecret` are available from EITHER source — the Device_Registration_File (loaded via `clientIdHash` per R1 criterion 4) OR directly from the Auth_Token_File (per R1 criterion 3) — THE Kiro_Auth_Manager SHALL set the auth type to AWS SSO OIDC and use the SSO OIDC refresh flow for all subsequent token refreshes. WHEN both sources provide `clientId`/`clientSecret`, the direct Auth_Token_File credentials SHALL take priority (override the Device_Registration_File credentials), matching the reference gateway load order where the companion file is loaded first and then direct fields overwrite.
2. WHEN the Kiro_Auth_Manager is initialized and `clientId` and `clientSecret` are not available from either source (neither the Device_Registration_File nor the Auth_Token_File), THE Kiro_Auth_Manager SHALL set the auth type to Kiro Desktop Auth and use the Kiro Desktop Auth refresh flow for all subsequent token refreshes.
3. WHEN a token refresh is needed and the auth type is AWS SSO OIDC, THE Kiro_Auth_Manager SHALL send an HTTP POST to `https://oidc.{region}.amazonaws.com/token` with a `Content-Type: application/json` header and a JSON body containing `{"grantType": "refresh_token", "clientId": "{clientId}", "clientSecret": "{clientSecret}", "refreshToken": "{current_refresh_token}"}` (camelCase parameter names, JSON payload).
4. WHEN a token refresh is needed and the auth type is Kiro Desktop Auth, THE Kiro_Auth_Manager SHALL send an HTTP POST to `https://prod.{region}.auth.desktop.kiro.dev/refreshToken` with a `Content-Type: application/json` header and a JSON body containing `{"refreshToken": "{current_refresh_token}"}`.
5. WHEN the SSO OIDC refresh endpoint returns a successful JSON response containing camelCase fields `accessToken`, `refreshToken`, and `expiresIn`, THE Kiro_Auth_Manager SHALL update the stored credentials with the new values and compute `expiresAt` per criterion 5a.
5a. WHEN the refresh response contains `expiresIn` (number of seconds), THE Kiro_Auth_Manager SHALL compute `expiresAt` as `new Date(Date.now() + expiresIn * 1000).toISOString()` and store this computed ISO 8601 string. THE persisted Auth_Token_File SHALL contain `expiresAt` as an ISO 8601 string, not the raw `expiresIn` value.
6. WHEN the Kiro Desktop Auth refresh endpoint returns a successful JSON response containing `accessToken`, `refreshToken`, and `expiresIn`, THE Kiro_Auth_Manager SHALL update the stored credentials with the new values and compute `expiresAt` per criterion 5a. WHEN the Kiro Desktop Auth refresh response also contains a `profileArn` field, THE Kiro_Auth_Manager SHALL update the stored `profileArn` value with the new value from the response.
7. WHEN the refresh succeeds, THE Kiro_Auth_Manager SHALL write the updated credentials back to the Auth_Token_File, preserving all other existing fields in the file.
8. IF the refresh request fails, THEN THE Kiro_Auth_Manager SHALL throw an error with the status code and response body from the failed refresh attempt.
9. WHILE a token refresh is in progress, THE Kiro_Auth_Manager SHALL prevent concurrent refresh attempts by reusing the pending refresh promise.
10. THE Kiro_Auth_Manager SHALL include a `User-Agent` header in the Kiro Desktop Auth refresh request matching the pattern `KiroIDE-{version}-{fingerprint}`.

### Requirement 4: Kiro API Request Headers

**User Story:** As a developer, I want the provider to send correctly formatted headers to the Kiro API, so that requests are accepted by the upstream service.

#### Acceptance Criteria

1. THE Kiro_Client SHALL include an `Authorization` header with the value `Bearer {accessToken}` on every request to the Kiro API.
2. THE Kiro_Client SHALL include a `Content-Type: application/json` header on every POST request.
3. THE Kiro_Client SHALL include an `x-amzn-codewhisperer-optout: true` header on every request.
4. THE Kiro_Client SHALL include a `User-Agent` header matching the pattern `aws-sdk-js/1.0.27 ua/2.1 os/{platform}#{version} lang/js md/nodejs#{nodeVersion} api/codewhispererstreaming#1.0.27 m/E KiroIDE-{kiroVersion}-{fingerprint}`.
5. THE Kiro_Client SHALL include an `x-amz-user-agent` header matching the pattern `aws-sdk-js/1.0.27 KiroIDE-{kiroVersion}-{fingerprint}`.
6. THE Kiro_Client SHALL include an `x-amzn-kiro-agent-mode: vibe` header on every request.
7. THE Kiro_Client SHALL include an `amz-sdk-invocation-id` header with a unique UUID v4 value per request.
8. THE Kiro_Client SHALL include an `amz-sdk-request: attempt=1; max=3` header on every request.
9. THE Kiro_Client SHALL derive the API host from the `region` field in the Auth_Token_File using the template `https://q.{region}.amazonaws.com`.

### Requirement 5: HTTP Retry Logic

**User Story:** As a developer, I want the provider to handle transient API failures gracefully, so that temporary issues do not cause immediate request failures.

#### Acceptance Criteria

1. THE Kiro_Client SHALL send the converted Kiro payload as an HTTP POST request to `https://q.{region}.amazonaws.com/generateAssistantResponse` with the headers specified in R4. This locks the endpoint path (`/generateAssistantResponse`) and method (`POST`) for the main API call.
2. WHEN the Kiro API returns HTTP 403, THE Kiro_Client SHALL trigger a token refresh via the Kiro_Auth_Manager and retry the request once with the new token.
3. WHEN the Kiro API returns HTTP 429, THE Kiro_Client SHALL wait using exponential backoff (base delay 1 second, multiplied by 2 per attempt) and retry up to 3 times.
4. WHEN the Kiro API returns HTTP 5xx, THE Kiro_Client SHALL wait using exponential backoff and retry up to 3 times.
5. IF all retry attempts are exhausted, THEN THE Kiro_Client SHALL throw a typed HTTP error (e.g., KiroHttpError) with the last received status code, headers, and error body. THE Kiro_Upstream_Provider SHALL catch this error and return a Canonical_ErrorResponse with the corresponding status code and error body.
6. THE Kiro_Client SHALL use a read timeout of 300 seconds for streaming requests to accommodate long model thinking pauses.
7. BEFORE sending any Kiro API request (`generateAssistantResponse` or `ListAvailableModels`), THE Kiro_Client SHALL obtain a valid access token by calling the Kiro_Auth_Manager's token accessor (e.g., `getAccessToken()`), which checks `isTokenExpiringSoon()` (per R2) and triggers a proactive refresh (per R3) under the pending promise (per R3 criterion 9) if the token is expiring soon or expired. The returned access token SHALL be used to build the `Authorization: Bearer {accessToken}` header for the request. This ensures tokens are refreshed proactively before API calls, not only reactively on 403 responses (criterion 2). The proactive check and the reactive 403 retry are complementary: the proactive check prevents most expired-token requests, while the 403 retry handles race conditions where a token expires between the check and the API call.


### Requirement 6: Canonical Request to Kiro Payload Conversion

**User Story:** As a developer, I want canonical requests to be correctly translated to the Kiro API format, so that the upstream service receives valid payloads.

#### Acceptance Criteria

1. THE Kiro_Payload_Converter SHALL convert a Canonical_Request into a Kiro `generateAssistantResponse` payload containing a `conversationState` object with `conversationId`, `currentMessage`, `chatTriggerType` set to `"MANUAL"`, and `history` (when non-empty). WHEN the post-split history is non-empty, THE Kiro_Payload_Converter SHALL include the repaired history entries at `conversationState.history` as an array of objects, each shaped as either `{ userInputMessage: {...} }` or `{ assistantResponseMessage: {...} }`. WHEN the post-split history is empty, THE Kiro_Payload_Converter SHALL omit the `history` field from `conversationState`.
1a. WHEN the auth type is Kiro Desktop Auth (per R3 criterion 2) AND the Kiro_Auth_Manager has a non-empty `profileArn` value, THE Kiro_Payload_Converter SHALL include `profileArn` as a top-level field in the `generateAssistantResponse` payload. WHEN the auth type is AWS SSO OIDC, THE Kiro_Payload_Converter SHALL NOT include `profileArn` in the payload, even if the Auth_Token_File contains one.
2. WHEN the post-split history is non-empty, THE Kiro_Payload_Converter SHALL prepend the `instructions` field to the content of the first `userInputMessage` in history, separated by `\n\n`.
2a. WHEN the post-split history is empty, THE Kiro_Payload_Converter SHALL prepend the `instructions` field to the `currentMessage.userInputMessage.content`, separated by `\n\n`.
2b. WHEN tool descriptions have been moved to documentation text (per criterion 10), THE Kiro_Payload_Converter SHALL append the tool documentation to the `instructions` text before embedding it into the message content.
3. THE Kiro_Payload_Converter SHALL process the Canonical_Request `input` array in the following complete processing order. Each step references the criterion that defines its detailed behavior:
    (1) **Validate server tools** (criterion 15): THE Kiro_Upstream_Provider validates that `Canonical_Request.tools` does not contain active server tools (`web_search`, `web_fetch`, `mcp`). IF any are present, THE Kiro_Upstream_Provider returns a Canonical_ErrorResponse with status 400 immediately. Do NOT proceed to subsequent steps. This validation runs BEFORE effectiveTools computation and is owned by the provider, not the converter.
    (2) **Compute effectiveTools** (criterion 18): The Kiro_Upstream_Provider computes `effectiveTools` by applying toolChoice handling to the remaining function tools, and passes the result to the Kiro_Payload_Converter as input. The converter receives `effectiveTools` as a parameter, not computing it itself.
    (3) **Tool-role extraction** (criterion 3a): Convert each message with role `tool` into a message with role `user` by extracting `function_call_output` content items as `toolResults`. After this step, there are no more `tool` role messages and `toolResults` are available for subsequent steps.
    (3.5) **Strip historical server-tool content** (criterion 15): Strip content items with types `server_tool`, `web_search`, `mcp_call`, or `mcp_call_output` from all messages in the input array. Log a warning for each stripped item. This runs after tool-role extraction (step 3) so that all messages are in user/assistant form, and before no-tools conversion (step 4) so that server-tool content is removed before any text conversion happens.
    (3.75) **Convert non-effective tool history for named toolChoice** (criterion 18): WHEN `effectiveTools` was narrowed to a single named tool, convert historical `toolUses` and `toolResults` for tools not in `effectiveTools` to text representation. This runs after tool-role extraction (step 3) and server-tool stripping (step 3.5) so that all messages are in user/assistant form with `toolResults` available, and before no-tools conversion (step 4) and the remaining repair pipeline (step 6) so that non-effective tool content is converted to text before merging and alternation repair.
    (4) **No-tools conversion** (criterion 17): IF `effectiveTools` is empty, convert all tool-related content (`function_call`, `toolResults`) to text representation across the ENTIRE input array. This runs AFTER tool-role extraction so that `toolResults` created in step (3) are available for text conversion.
    (5) **Validate tool names** (criterion 21a): IF any function tool name in `effectiveTools` exceeds 64 characters, throw a typed error with status 400 listing the problematic tool names and their lengths.
    (6) **Remaining repair pipeline** (criterion 3a-remaining, steps i-iv): Normalize unknown roles, merge adjacent same-role messages, ensure user-first, ensure alternating. Apply to the ENTIRE `Canonical_Request.input` array before splitting.
    (7) **Split** (criterion 3b): All messages except the last become history; the last message becomes the source for `currentMessage`.
    (8) **Derive currentMessage** (criteria 12-14, criterion 3c): Apply assistant-last → `"Continue"`, tool-last → toolResults, empty → `"Continue"` rules.
    (9) **Embed system prompt** (criteria 2/2a/2b, criterion 3d): Prepend to first history message when history is non-empty, or prepend to currentMessage when history is empty.
    (10) **Validate orphaned toolResults** (criteria 13a/13b, criterion 3e): Check both history and currentMessage for tool results without matching tool calls.
    (10.5) **Ensure non-empty content** (criterion 14a): Ensure every `userInputMessage.content` and `assistantResponseMessage.content` in history is non-empty, using `"(empty)"` as fallback.
    (10.75) **Inject thinking tags** (criterion 22): WHEN `reasoningEffort` is present, prepend thinking mode tags to `currentMessage` content. This runs AFTER system prompt embedding (step 9) and ensure non-empty content (step 10.5), and BEFORE building the Kiro payload (step 11), so that the tags are included in the final `currentMessage` content and counted toward the payload size limit.
    (11) **Build Kiro payload** with tools from `effectiveTools` placed at `currentMessage.userInputMessage.userInputMessageContext.tools` (criterion 8a).
    (12) **Measure payload size and trim** (criterion 19): If payload exceeds 400 KB, trim oldest history pairs and re-validate orphaned toolResults.
3a. In step (3), THE Kiro_Payload_Converter SHALL perform tool-role extraction on the ENTIRE `Canonical_Request.input` array. THE Kiro_Payload_Converter SHALL convert each message with role `tool` into a message with role `user` by extracting `function_call_output` content items as `toolResults` using the exact Kiro wire shape: `{ toolUseId: "{call_id}", content: [{ text: "{output}" }], status: "success" }` (where `content` is an array of `{ text: string }` objects, not a plain string). WHEN the `output` field of a `function_call_output` content item is empty or absent, THE Kiro_Payload_Converter SHALL use `"(empty result)"` as the text value in the `content` array. This prevents Kiro API validation failures for empty tool results. Any remaining non-tool content items become the user message `content`. If no non-tool content remains, the content SHALL be empty (to be handled by the `"Continue"` fallback later). After this step, there are no more `tool` role messages and `toolResults` are available for the no-tools conversion in step (4).
3a-remaining. In step (6), THE Kiro_Payload_Converter SHALL apply the remaining repair pipeline steps to the ENTIRE `Canonical_Request.input` array before splitting. The repair steps SHALL be applied in order: (i) normalize unknown roles (any role that is not `user` or `assistant`) to `user` — this only affects truly unknown roles like `developer`, `system`, etc., since `tool` roles have already been converted in step (3); (ii) merge adjacent messages with the same role by concatenating content and merging tool_calls and tool_results; (iii) ensure the first message is a `userInputMessage` — WHEN the first message is an `assistantResponseMessage`, THE Kiro_Payload_Converter SHALL prepend a synthetic `userInputMessage` with content `"(empty)"`; (iv) ensure alternating `userInputMessage` / `assistantResponseMessage` roles — WHEN two consecutive messages have the same role after merging, THE Kiro_Payload_Converter SHALL insert a synthetic message of the opposite role with content `"(empty)"` between them.
3b. In step (7), THE Kiro_Payload_Converter SHALL split the repaired message array: all messages except the last become history, the last message becomes the source for `currentMessage`.
3c. In step (8), THE Kiro_Payload_Converter SHALL derive `currentMessage.userInputMessage` from the last message using the rules in criteria 12-14 (assistant-last → add to history and use `"Continue"`, tool-last → toolResults, empty → `"Continue"`).
3d. In step (9), THE Kiro_Payload_Converter SHALL embed the system prompt using criteria 2/2a/2b (based on post-split history length: prepend to first history message when history is non-empty, or prepend to currentMessage when history is empty).
3e. In step (10), THE Kiro_Payload_Converter SHALL validate orphaned toolResults in both history (per criterion 13a) and currentMessage (per criterion 13b).
4. THE Kiro_Payload_Converter SHALL convert each Canonical_InputMessage with role `user` into a Kiro `userInputMessage` containing `content`, `modelId`, and `origin` set to `"AI_EDITOR"`, plus an optional `userInputMessageContext` for tool results.
5. WHEN a Canonical_InputMessage with role `assistant` contains `function_call` content items (type `function_call` with `call_id`, `name`, and `arguments` fields), THE Kiro_Payload_Converter SHALL convert each into a Kiro `toolUse` structure with `toolUseId` mapped from `call_id`, `name`, and `input` parsed from `arguments`.
6. WHEN a user message (converted from role `tool` per step (3)) carries `toolResults`, THE Kiro_Payload_Converter SHALL include these in the Kiro `userInputMessage.userInputMessageContext.toolResults` structure. Each tool result SHALL use the exact Kiro wire shape: `{ toolUseId: "{call_id}", content: [{ text: "{output}" }], status: "success" }`. The `content` field MUST be an array of `{ text: string }` objects, not a plain string. Assigning `output` directly as a string to `content` SHALL be treated as a bug, because the Kiro API validates that `content` is an array of text objects.
7. THE Kiro_Payload_Converter SHALL convert each Canonical_InputMessage with role `assistant` that contains text content into a Kiro `assistantResponseMessage` containing `content`.
8. WHEN the Canonical_Request contains function tools (items in the `tools` array with `type: "function"`), THE Kiro_Payload_Converter SHALL convert each into a Kiro `toolSpecification` with `name` from the tool's `name` field, `description` from the tool's `description` field, and `inputSchema` wrapped in the Kiro wire shape. The raw schema SHALL be read from the tool's `parameters` field (the canonical function tool schema field); if `parameters` is absent, THE Kiro_Payload_Converter SHALL fall back to the `input_schema` field, then to `{ type: "object", properties: {} }` as default. After sanitization (per criterion 9), the schema SHALL be assigned as `inputSchema: { json: sanitizedSchema }`. The full Kiro tool structure is: `{ toolSpecification: { name, description, inputSchema: { json: sanitizedSchema } } }`. Assigning the raw schema directly to `inputSchema` without the `{ json: ... }` wrapper SHALL be treated as a bug, because the Kiro API rejects tool definitions where `inputSchema` is not wrapped.
8a. THE Kiro_Payload_Converter SHALL place the converted `effectiveTools` array at `conversationState.currentMessage.userInputMessage.userInputMessageContext.tools` in the Kiro payload. This is the same `userInputMessageContext` object that may also contain `toolResults` (per criterion 6). WHEN `effectiveTools` is empty (no function tools or `toolChoice` is `none`), THE Kiro_Payload_Converter SHALL omit the `tools` field from `userInputMessageContext` entirely. Placing tools at any other location in the payload (e.g., top-level or directly on `conversationState`) SHALL be treated as a bug, because Kiro expects tools at this specific path for function-tool call loops.
9. THE Kiro_Payload_Converter SHALL sanitize tool input schemas by removing empty `required` arrays and `additionalProperties` fields that the Kiro API rejects.
10. WHEN a tool description exceeds 10000 characters, THE Kiro_Payload_Converter SHALL move the full description to the system prompt and replace the tool description with a reference string.
11. THE Kiro_Payload_Converter SHALL generate a unique random UUID v4 per upstream request as the `conversationId`.
12. WHEN the last Canonical_InputMessage has role `assistant`, THE Kiro_Payload_Converter SHALL add the assistant message to history and create a synthetic user `currentMessage` with `content` set to `"Continue"`.
13. WHEN the last message in the repaired array is a user message carrying `toolResults` (converted from role `tool` per step (3)), THE Kiro_Payload_Converter SHALL place the tool results into `currentMessage.userInputMessage.userInputMessageContext.toolResults` and use `"Continue"` as the `content` fallback when no text content is present.
13a. WHEN a user message in history contains tool results but the preceding assistant message does not contain matching `toolUses` (by `toolUseId`), THE Kiro_Payload_Converter SHALL convert the orphaned tool results to text representation and append to the user message content, removing them from `userInputMessageContext.toolResults`. The text representation SHALL be `[Tool Result ({toolUseId})]\n{flattenedText}` where `flattenedText` is produced by extracting the `text` field from each object in the `content` array (Kiro wire shape `[{ text: "..." }]`) and joining with newlines. WHEN `flattenedText` would be empty (e.g., all `content` entries have empty `text` values), THE Kiro_Payload_Converter SHALL use `"(empty result)"` as the `flattenedText` value. Implementations SHALL NOT stringify the `content` array directly. This prevents Kiro API validation failures for tool results without matching tool calls.
13b. WHEN the `currentMessage` contains tool results in `userInputMessageContext.toolResults` but the last `assistantResponseMessage` in history does not contain matching `toolUses` (by `toolUseId`), THE Kiro_Payload_Converter SHALL convert the orphaned tool results to text representation and append them to the existing `currentMessage` content, removing them from `userInputMessageContext.toolResults`. If `currentMessage` already has content (text or embedded instructions), the orphaned tool result text SHALL be appended with `\n\n` separator, not replace it. The text representation SHALL follow the same flattening rule as criterion 13a: `[Tool Result ({toolUseId})]\n{flattenedText}` with `flattenedText` extracted from `content[].text` and joined with newlines. WHEN `flattenedText` would be empty, THE Kiro_Payload_Converter SHALL use `"(empty result)"` as the `flattenedText` value, consistent with criterion 13a. This handles tool-result-only turns or truncated context where currentMessage toolResults have no preceding assistant toolUses match.
14. WHEN the derived `currentMessage` content would be empty, THE Kiro_Payload_Converter SHALL use `"Continue"` as the content value.
14a. AFTER the repair pipeline (steps 1-6 of criterion 3) and BEFORE building the Kiro payload (step 11), THE Kiro_Payload_Converter SHALL ensure every `userInputMessage.content` and `assistantResponseMessage.content` across ALL history messages is non-empty. WHEN a history message has empty content (which can occur after: tool-result-only turns where content was extracted to `toolResults`, stripped server-tool content items per criterion 15 step 3.5, or synthetic repair messages), THE Kiro_Payload_Converter SHALL use `"(empty)"` as the content value. This is separate from criterion 14 which handles the `currentMessage` specifically with `"Continue"`. The Kiro API rejects empty `content` in both `userInputMessage` and `assistantResponseMessage` regardless of position in the conversation.
15. THE Kiro_Upstream_Provider SHALL validate that `Canonical_Request.tools` does not contain active server tools BEFORE computing `effectiveTools`. IF any tools have `type` equal to `web_search`, `web_fetch`, or `mcp`, THE Kiro_Upstream_Provider SHALL return a Canonical_ErrorResponse with status 400 and a message indicating that server tools (`web_search`, `web_fetch`, `mcp`) are not supported by the Kiro upstream provider in phase 1. This validation runs FIRST in the processing pipeline (step 1 of criterion 3), BEFORE effectiveTools computation (criterion 18). If ANY active server tools are present in `Canonical_Request.tools`, the request is rejected immediately — the provider does not proceed to effectiveTools computation, payload conversion, or any subsequent steps. The Kiro_Payload_Converter MAY keep a defensive assertion for server tools, but the Kiro_Upstream_Provider is the required validation owner. Historical server-tool content in input messages is handled separately in step (3.5) of the processing pipeline (criterion 3): content items with types `server_tool`, `web_search`, `mcp_call`, or `mcp_call_output` in input messages SHALL be stripped as historical references (they are not active tool requests). THE Kiro_Payload_Converter SHALL log a warning when stripping such historical content blocks. This stripping runs after tool-role extraction (step 3) and before no-tools conversion (step 4), ensuring server-tool content is removed before any text conversion happens.
16. FOR ALL valid Canonical_Request objects that do not contain active server tools (`web_search`, `web_fetch`, `mcp`), that have a non-empty `effectiveTools` set, where `toolChoice` is NOT a named tool object (i.e., `toolChoice` is `auto`, `required`, `none`, or absent), and after stripping historical `server_tool`/`web_search`/`mcp_call`/`mcp_call_output` content blocks, converting to a Kiro payload and then extracting the message content SHALL preserve the original text content and function tool call structure (round-trip content preservation property). This property explicitly excludes the no-tools case (criterion 17), because the no-tools transform intentionally converts tool-related content to text representation, which is a lossy transformation. This property also excludes named toolChoice (criterion 18, step 3.75), because named toolChoice converts non-effective tool history to text representation, which is also a lossy transformation.
17. WHEN `effectiveTools` (as computed by criterion 18) is empty, THE Kiro_Payload_Converter SHALL convert all tool-related content to text representation. This transform runs as step (4) of the processing pipeline (criterion 3), AFTER tool-role extraction (step 3) has already converted `role: "tool"` messages to `role: "user"` messages with `toolResults`. Because tool-role extraction has already run, `toolResults` are available for text conversion at this point. This transform SHALL apply to the ENTIRE input array (before the remaining repair pipeline steps and the split into history + currentMessage per criterion 3b), not just history. This ensures both history messages and the message that becomes `currentMessage` have tool content converted to text when no tools are defined. Specifically: `function_call` items become `[Tool: {name} ({call_id})]\n{arguments}` where `arguments` is the string value of the `arguments` field (already a JSON string from the canonical `function_call` content item) — if `arguments` was parsed to an object during `toolUse` conversion (criterion 5), it SHALL be re-serialized via `JSON.stringify()` for the text representation. `toolResults` become `[Tool Result ({toolUseId})]\n{flattenedText}` where `flattenedText` is produced by extracting the `text` field from each object in the `content` array (which is in Kiro wire shape `[{ text: "..." }]` after tool-role extraction in step 3) and joining them with newlines. WHEN `flattenedText` would be empty, THE Kiro_Payload_Converter SHALL use `"(empty result)"` as the `flattenedText` value, consistent with criteria 3a and 13a/13b. Implementations SHALL NOT stringify the `content` array directly (e.g., producing `[{"text":"output"}]`); they MUST flatten it to plain text. The converted text SHALL be appended to the message content, and `toolUses`/`toolResults` SHALL be removed from the Kiro payload. After this transform, no message in the array SHALL contain `userInputMessageContext.toolResults` or `assistantResponseMessage.toolUses`. This prevents Kiro API rejection of `toolResults` without corresponding tool definitions, including the case where a tool-result-only turn becomes `currentMessage`. Note: `effectiveTools` is empty when the original `tools` array is absent/empty, OR when `toolChoice` is `none` (criterion 18 strips all tools). The case where all tools were server tools cannot reach this point because server-tool validation (criterion 15, step 1) rejects the request with 400 before effectiveTools computation.
18. THE Kiro_Upstream_Provider SHALL compute `effectiveTools` — the final set of function tools that will be sent to Kiro — from `Canonical_Request.tools` and `toolChoice` before calling the Kiro_Payload_Converter. By this point, server-tool validation (criterion 15, step 1) has already passed, meaning `Canonical_Request.tools` contains only function tools or is empty. The processing order SHALL be: (a) server tools have already been rejected by criterion 15 in step 1 — no server tools remain in the tools array at this point; (b) apply toolChoice handling per this criterion; (c) the resulting set is `effectiveTools`; (d) criterion 17 (step 4) runs whenever `effectiveTools` is empty. WHEN the Canonical_Request contains a `toolChoice` field, THE Kiro_Upstream_Provider SHALL handle it as follows, because the Kiro `generateAssistantResponse` API does not support tool choice constraints. WHEN `toolChoice` is set to `none` (meaning the model should not use tools), THE Kiro_Upstream_Provider SHALL set `effectiveTools` to empty to approximate the no-tool-use behavior, which triggers criterion 17's no-tools conversion of all tool-related content to text. WHEN `toolChoice` is `auto`, the tools SHALL be included normally in `effectiveTools` and the model will decide tool usage. WHEN `toolChoice` is `required`, THE Kiro_Upstream_Provider SHALL log a warning that Kiro does not support forced tool selection, and include all tools in `effectiveTools` normally. This is a known limitation — Kiro may return text-only `end_turn` responses even when the client requested forced tool use via `required`. The `required` toolChoice is commonly used by Claude Code and rejecting it with 400 would break basic functionality. The model usually does call tools even without forced selection, so including all tools normally is a reasonable approximation. WHEN `toolChoice` is a named tool object (forcing a specific tool), THE Kiro_Upstream_Provider SHALL log a warning that Kiro does not support forced tool selection, and narrow `effectiveTools` to contain ONLY the named tool (if the named tool exists in the original function tools array). This approximates forced selection by removing other tools from the payload, making it more likely the model uses the specified tool. WHEN `toolChoice` is a named tool object AND the conversation history contains `toolUses` for tools not in the narrowed `effectiveTools`, THE Kiro_Upstream_Provider SHALL convert those historical non-effective `toolUses` and their corresponding `toolResults` to text representation, preserving only the named tool's `toolUses`/`toolResults` in structured form. The text conversion SHALL use the same format as criterion 17's no-tools conversion: `function_call` items become `[Tool: {name} ({call_id})]\n{arguments}`, and `toolResults` become `[Tool Result ({toolUseId})]\n{flattenedText}` (with `flattenedText` extracted from `content[].text` and joined with newlines, using `"(empty result)"` when empty). This conversion is a defensive measure that prevents potential Kiro validation failures where historical tool call names do not match the current tool definitions. This conversion is step (3.75) of the processing pipeline (criterion 3). It runs after tool-role extraction (step 3) and server-tool stripping (step 3.5), and before no-tools conversion (step 4) and the remaining repair pipeline (step 6), so that non-effective tool content is converted to text before merging and alternation repair. The named tool's own `toolUses`/`toolResults` SHALL remain in structured form. IF the named tool specified in `toolChoice` is not found in the `Canonical_Request.tools` array, THE Kiro_Upstream_Provider SHALL return a Canonical_ErrorResponse with status 400 and a message indicating the named tool was not found in the provided tools. The `toolChoice` field itself SHALL NOT be included in the Kiro payload. The Kiro_Payload_Converter receives `effectiveTools` as input and does not compute it.
18a. THE Kiro_Upstream_Provider SHALL compute `effectiveTools` once (per criterion 18) and pass the same `effectiveTools` set to both the Kiro_Payload_Converter (for payload construction and tool conversion in criteria 8/8a/17) and the Kiro_Stream_Parser or response collector (for bracket-style tool call extraction in R20 criterion 4). This ensures the bracket parser has access to the same `effectiveTools` set used for payload conversion, preventing mismatches where the parser might accept or reject tool names differently from the payload. The Kiro_Upstream_Provider is the natural owner of this coordination because it orchestrates both payload conversion and response parsing.
19. THE Kiro_Payload_Converter SHALL measure the serialized JSON payload size after conversion. WHEN the payload exceeds 400000 bytes (400 KB, a conservative limit below the Kiro gateway's approximately 600 KB maximum), THE Kiro_Payload_Converter SHALL enter a trim-and-revalidate loop: (a) trim the oldest message pair (user + assistant) from the beginning of history; (b) re-run orphaned toolResults validation (per criteria 13a/13b) on the trimmed history and currentMessage, because trimming may remove an assistant message containing toolUses while leaving a subsequent user message with toolResults that reference the removed toolUses; (c) re-embed the system prompt and tool documentation (per criteria 2/2a/2b) into the new first history message (or currentMessage if history is now empty), because trimming may have removed the original instruction-bearing message; (d) re-measure the serialized JSON payload size of the FINAL payload (after orphan repair and system prompt re-embedding). WHEN the re-measured payload still exceeds 400000 bytes, THE Kiro_Payload_Converter SHALL repeat steps (a)-(d) — trimming more history pairs — because post-trim repairs (orphan-to-text conversion and system prompt re-embedding) can increase payload size, potentially pushing it back over the limit. The loop SHALL continue until either the final payload fits within 400000 bytes or all removable history pairs are exhausted. After trimming, the repair pipeline's user-first and alternating constraints SHALL still hold. IF all removable history pairs are exhausted and the final re-measured payload still exceeds the limit, THE Kiro_Payload_Converter SHALL throw an error that the Kiro_Upstream_Provider maps to a Canonical_ErrorResponse with status 413 and a message indicating the payload is too large. THE Kiro_Payload_Converter SHALL log a warning indicating how many history entries were trimmed and the original payload size.
20. WHEN a Canonical_InputMessage with role `user` contains `input_image` content items (with an `image_url` field containing a data URL in the format `data:{mediaType};base64,{base64Data}`), THE Kiro_Payload_Converter SHALL extract the base64 data and media type, convert to the Kiro image format `{ format: "{ext}", source: { bytes: "{base64}" } }` (where `{ext}` is derived from the media type, e.g., `image/png` → `png`, `image/jpeg` → `jpeg`), and place the result in `userInputMessage.images` (an array on the message, not in `userInputMessageContext`). WHEN a content item has type `input_file`, THE Kiro_Payload_Converter SHALL handle it based on the content source: (a) WHEN `input_file` has `file_data` containing a data URL with a text-based media type (e.g., `text/plain`, `text/markdown`, `application/json`), THE Kiro_Payload_Converter SHALL decode the base64 content and include it as text in the message content with a `Document: {filename}` prefix; (b) WHEN `input_file` has `file_data` with a binary media type (e.g., `application/pdf`, `image/*`), THE Kiro_Payload_Converter SHALL include a text placeholder in the message content indicating the skipped content (e.g., `[Unsupported: binary document "{filename}" skipped — Kiro does not support binary file attachments]`) and log a warning; (c) WHEN `input_file` has `file_url` or `file_id`, THE Kiro_Payload_Converter SHALL include a text placeholder in the message content indicating the skipped content (e.g., `[Unsupported: document "{filename}" skipped — Kiro does not support URL-based or file-ID-based file attachments]`) and log a warning. WHEN an `input_image` content item contains a URL-based image (not a data URL), THE Kiro_Payload_Converter SHALL include a text placeholder in the message content (e.g., `[Unsupported: URL-based image skipped — Kiro only supports base64 data URL images]`), log a warning, and skip the image conversion. These placeholders make the omission visible to the model so it can acknowledge the limitation rather than silently ignoring the content.
21. THE Kiro_Payload_Converter SHALL validate tool name lengths before converting function tools to Kiro `toolSpecification` format.
22. WHEN the Canonical_Request contains a `reasoningEffort` field, THE Kiro_Payload_Converter SHALL inject thinking mode tags into the last user message content (`currentMessage`) to enable fake reasoning (step 10.75 of criterion 3). Specifically, THE Kiro_Payload_Converter SHALL prepend `<thinking_mode>enabled</thinking_mode>\n<max_thinking_length>{budget}</max_thinking_length>\n` to the `currentMessage` content, where `{budget}` is derived from `reasoningEffort` (e.g., `low` → 4000, `medium` → 8000, `high` → 16000, `xhigh` → 32000). WHEN `reasoningEffort` is absent or not recognized, THE Kiro_Payload_Converter SHALL NOT inject thinking tags. This enables the Kiro model to produce `<thinking>...</thinking>` blocks in its response, which are then parsed by R10. **Known limitation:** In the existing Claude inbound provider, the `reasoningEffort` field in `Canonical_Request` is only populated from `body.output_config?.effort` (the Claude `output_config.effort` mechanism). The Claude `thinking` configuration (`body.thinking: { type: "enabled", budget_tokens: N }`) is a separate mechanism that is NOT mapped into `Canonical_Request` by the existing `claudeToCanonicalRequest` function. Therefore, thinking tag injection will only activate when a client sets `output_config.effort`, not when a client uses `thinking: { type: "enabled", budget_tokens: N }`. Requests using the `thinking` configuration will be processed as normal (non-thinking) requests and return plain text 200 responses. This is a known limitation of the existing Claude inbound provider's canonical mapping, not specific to the Kiro upstream provider. A subsequent phase SHALL update the Claude inbound provider to map `body.thinking` into `Canonical_Request` (e.g., by adding a `thinking` field to `Canonical_Request` or mapping `thinking.budget_tokens` to `reasoningEffort`), enabling thinking tag injection for both mechanisms.
21a. BEFORE converting function tools in `effectiveTools` to Kiro `toolSpecification` format (step 5 of criterion 3), THE Kiro_Payload_Converter SHALL validate that each tool name does not exceed 64 characters. IF any tool name exceeds 64 characters, THE Kiro_Payload_Converter SHALL throw a typed error (e.g., `ToolNameTooLongError`) with a message listing the problematic tool names and their lengths. THE Kiro_Upstream_Provider SHALL catch this error and return a Canonical_ErrorResponse with status 400 and the error message. This prevents vague upstream 400 errors from the Kiro API which rejects `toolSpecification` names over 64 characters.

### Requirement 7: AWS Event-Stream Parsing

**User Story:** As a developer, I want the provider to parse the Kiro binary event-stream format, so that streaming responses can be processed incrementally.

#### Acceptance Criteria

1. THE Kiro_Stream_Parser SHALL parse binary chunks from the Kiro API response by detecting JSON event boundaries delimited by `:message-type` markers.
2. WHEN a chunk contains a `{"content": "..."}` event, THE Kiro_Stream_Parser SHALL emit a content event with the text value.
3. WHEN a chunk contains a `{"name": "...", "toolUseId": "...", "input": "..."}` event, THE Kiro_Stream_Parser SHALL begin accumulating a tool call with the given name and ID.
4. WHEN a chunk contains a `{"input": "..."}` continuation event, THE Kiro_Stream_Parser SHALL append the input data to the current tool call arguments.
5. WHEN a chunk contains a `{"stop": true}` event, THE Kiro_Stream_Parser SHALL finalize the current tool call and add it to the completed tool calls list.
6. WHEN a chunk contains a `{"contextUsagePercentage": N}` event, THE Kiro_Stream_Parser SHALL emit a context usage event with the percentage value.
7. THE Kiro_Stream_Parser SHALL handle chunks that split JSON events across multiple network packets by buffering incomplete data.
8. THE Kiro_Stream_Parser SHALL deduplicate consecutive identical content events.


### Requirement 8: Streaming Response to Canonical Events

**User Story:** As a developer, I want the Kiro streaming response to be converted to canonical stream events, so that inbound providers can format responses without knowing about the Kiro wire format.

#### Acceptance Criteria

1. THE Kiro_Stream_Parser SHALL produce a Canonical_StreamResponse with an async iterable of Canonical_Event objects.
2. WHEN a content event is parsed, THE Kiro_Stream_Parser SHALL yield a `text_delta` Canonical_Event with the content text.
3. WHEN a tool call is finalized, THE Kiro_Stream_Parser SHALL yield a `tool_call_done` Canonical_Event with the call ID, name, and accumulated arguments.
4. WHEN the stream ends, THE Kiro_Stream_Parser SHALL yield a `message_stop` Canonical_Event with the appropriate stop reason: `end_turn` for normal completion, `tool_use` when tool calls are present, or `max_tokens` when the stream was truncated.
5. THE Kiro_Stream_Parser MAY yield `content_block_start` and `content_block_stop` events for future use, but the existing `claudeCanonicalStreamResponse` does not consume them. The `claudeCanonicalStreamResponse` function manages its own `contentIndex` and block boundaries internally based on `text_delta`, `thinking_delta`, `thinking_signature`, `tool_call_done`, `server_tool_block`, `usage`, `message_stop`, `completion`, and `error` events. The canonical events that the Kiro_Stream_Parser MUST yield are: `text_delta` (when text content is present), `usage`, and `message_stop`. Additionally, `thinking_delta` and `thinking_signature` MUST be yielded when thinking content is detected, and `tool_call_done` MUST be yielded when tool calls are present.
6. THE Kiro_Stream_Parser SHALL yield a `usage` event with estimated input and output token counts.

### Requirement 9: Non-Streaming Response Collection

**User Story:** As a developer, I want the provider to support non-streaming responses, so that clients requesting synchronous responses receive a complete Canonical_Response.

#### Acceptance Criteria

1. WHEN a non-streaming request is made, THE Kiro_Stream_Parser SHALL consume the entire event stream and return a single Canonical_Response.
2. THE Canonical_Response SHALL contain the accumulated text content, all finalized tool calls, the stop reason, and usage information.
3. THE Canonical_Response SHALL include a generated response ID in the format `resp_{uuid}`.
4. WHEN the Canonical_Response contains one or more `Canonical_ToolCallBlock` entries in its `content` array, THE `stopReason` SHALL be set to `tool_use`. This applies to both structured tool calls from Kiro events and bracket-style tool calls extracted per R20. This ensures Claude Messages parity where `stop_reason: tool_use` signals the client to process tool calls, preventing broken tool loops caused by `stop_reason: end_turn` on responses containing `tool_use` blocks.

### Requirement 10: Thinking Block Extraction

**User Story:** As a developer, I want the provider to extract thinking/reasoning blocks from Kiro responses, so that clients requesting extended thinking receive structured reasoning content.

#### Acceptance Criteria

1. WHEN the Kiro response content begins with a `<thinking>` or `<think>` tag, THE Kiro_Stream_Parser SHALL extract the content between the opening and closing tags as thinking content.
2. THE Kiro_Stream_Parser SHALL yield the thinking block event sequence in the following locked order: `content_block_start` with blockType `thinking` → `thinking_delta` events for the thinking content → `thinking_signature` (per criterion 3) → `content_block_stop`. The `thinking_signature` event MUST be yielded AFTER all `thinking_delta` events and BEFORE `content_block_stop`, because the Claude formatter (`claudeCanonicalStreamResponse`) emits `signature_delta` when it receives `thinking_signature` and then closes the thinking block on `content_block_stop`.
3. THE Kiro_Stream_Parser SHALL generate a thinking signature in the format `sig_{uuid_hex}` for each thinking block.
4. WHEN the thinking block ends, THE Kiro_Stream_Parser SHALL treat all subsequent content as regular text content.
5. THE Kiro_Stream_Parser SHALL buffer the initial response content (up to 30 characters) to detect thinking tags that may be split across chunks.

### Requirement 11: Model Name Normalization

**User Story:** As a developer, I want model names from various client formats to be normalized to the Kiro API format, so that clients can use familiar model names.

#### Acceptance Criteria

1. WHEN a model name uses dash-separated version numbers (e.g., `claude-sonnet-4-5`), THE Kiro_Model_Resolver SHALL normalize the minor version separator to a dot (e.g., `claude-sonnet-4.5`).
2. WHEN a model name includes a date suffix (e.g., `claude-sonnet-4-20250514`), THE Kiro_Model_Resolver SHALL strip the date suffix.
3. WHEN a model name includes a `-latest` suffix, THE Kiro_Model_Resolver SHALL strip the suffix.
4. WHEN a model name uses legacy format (e.g., `claude-3-7-sonnet`), THE Kiro_Model_Resolver SHALL normalize to dot format (e.g., `claude-3.7-sonnet`).
5. WHEN a model name is not recognized, THE Kiro_Model_Resolver SHALL pass it through unchanged to the Kiro API.
6. FOR ALL model names, normalizing twice SHALL produce the same result as normalizing once (idempotence property).

### Requirement 12: Model Listing

**User Story:** As a developer, I want the provider to list available models from the Kiro API, so that clients can discover which models are available.

#### Acceptance Criteria

1. WHEN a model list is requested, THE Kiro_Client SHALL send a GET request to `https://q.{region}.amazonaws.com/ListAvailableModels` with query parameter `origin=AI_EDITOR` and appropriate authorization headers. WHEN the auth type is Kiro Desktop Auth (per R3 criterion 2) AND the Kiro_Auth_Manager has a non-empty `profileArn` value, THE Kiro_Client SHALL include `profileArn={profileArn}` as an additional query parameter. WHEN the auth type is AWS SSO OIDC, THE Kiro_Client SHALL NOT include the `profileArn` query parameter, even if the Auth_Token_File contains one, because sending `profileArn` for SSO OIDC can cause 403 errors.
2. THE Kiro_Upstream_Provider SHALL cache the model list for 3600 seconds to avoid excessive API calls.
3. THE Kiro_Upstream_Provider SHALL include hidden models (models not returned by the API but known to be functional) in the model list.
4. IF the ListAvailableModels API is unreachable, THEN THE Kiro_Upstream_Provider SHALL return a fallback list of known models.
5. THE Kiro_Upstream_Provider SHALL expose a `listModels(): Promise<string[]>` method (not part of the Upstream_Provider interface). THE Bootstrap SHALL inject this method as the `modelResolver` callback into the Claude_Inbound_Provider constructor, following the same pattern used by the existing `claudeSettingsModelResolver` default. THE Codex upstream provider SHALL continue using `claudeSettingsModelResolver` unchanged.
6. Phase 1 SHALL accept that `/v1/models/:model_id` may return 404 for Kiro models that are not in the static Model_Catalog. This is a known limitation because the Kiro `listModels()` method returns models dynamically from the API, and these models may not have corresponding entries in the static Model_Catalog used by the single-model route. This limitation SHALL be tracked for a subsequent phase, which may register Kiro model metadata in the Model_Catalog or make the single-model route resolver-aware.


### Requirement 13: Token Count Fallback

**User Story:** As a developer, I want the provider to estimate token counts locally, so that usage information is available even though the Kiro API has no dedicated token counting endpoint.

#### Acceptance Criteria

1. THE Kiro_Upstream_Provider SHALL use a local tokenizer (gpt-tokenizer) to estimate output token counts from response text.
2. WHEN the Kiro API returns a `contextUsagePercentage` value, THE Kiro_Upstream_Provider SHALL calculate input tokens as `(contextUsagePercentage / 100) * maxInputTokens - outputTokens`, clamped to a minimum of 0.
3. WHEN model token limits are unknown, THE Kiro_Upstream_Provider SHALL use a default `maxInputTokens` of 200000.
4. IF no `contextUsagePercentage` is available, THEN THE Kiro_Upstream_Provider SHALL estimate input tokens from the request messages using the local tokenizer.

### Requirement 14: Bootstrap Provider Selection

#### Acceptance Criteria

1. WHEN the `UPSTREAM_PROVIDER` environment variable is set to `kiro`, THE Bootstrap SHALL instantiate the Kiro_Upstream_Provider.
2. WHEN the `UPSTREAM_PROVIDER` environment variable is set to `codex` or is not set, THE Bootstrap SHALL instantiate the Codex_Upstream_Provider (existing behavior).

4. THE Bootstrap SHALL return a synthetic state file path (e.g., `~/.codex2claudecode/kiro-state.json`) as the `authFile` value for Kiro mode, so that `requestLogFilePath(authFile)` (which uses `path.dirname(authFile)`) resolves to `~/.codex2claudecode/` for request logs. The `authFile` value SHALL be a file path (not a directory path), because `requestLogFilePath` calls `path.dirname(authFile)`. The actual Kiro credential source path (`~/.aws/sso/cache/kiro-auth-token.json`) is internal to Kiro_Upstream_Provider and not exposed as `authFile`.

### Requirement 15: Provider Architecture Compliance

#### Acceptance Criteria

THE Kiro upstream provider SHALL be implemented in `src/upstream/kiro/` with separate files for: `index.ts` (provider class), `client.ts` (HTTP client), `auth.ts` (credential management), `payload.ts` (request conversion), `parse.ts` (response parsing), `types.ts` (Kiro-specific types), and `constants.ts` (endpoints and configuration).

### Requirement 16: Credential File Write-Back

**User Story:** As a developer, I want refreshed tokens to be persisted to disk, so that subsequent startups do not require re-authentication.

#### Acceptance Criteria

1. WHEN a token refresh succeeds, THE Kiro_Auth_Manager SHALL write the updated `accessToken`, `refreshToken`, and `expiresAt` back to the Auth_Token_File.

3. THE Kiro_Auth_Manager SHALL create parent directories if they do not exist before writing the file.
4. FOR ALL valid Auth_Token_File contents, reading the file, performing a refresh, writing back, and reading again SHALL produce a file containing all original non-token fields unchanged (round-trip file preservation property).

### Requirement 17: Kiro Payload Tool Schema Sanitization

**User Story:** As a developer, I want tool schemas to be sanitized before sending to the Kiro API, so that the API does not reject payloads with unsupported schema fields.

#### Acceptance Criteria

### Requirement 18: Error Response Handling

**User Story:** As a developer, I want Kiro API errors to be translated to canonical error responses, so that inbound providers can format appropriate error messages for clients.

#### Acceptance Criteria

1. WHEN the Kiro API returns a non-200 response that is not retryable, THE Kiro_Upstream_Provider SHALL return a Canonical_ErrorResponse with the HTTP status code, response headers, and error body.
2. WHEN a network timeout occurs after all retries are exhausted, THE Kiro_Upstream_Provider SHALL return a Canonical_ErrorResponse with status 504.
3. WHEN the Auth_Token_File cannot be read or tokens cannot be refreshed, THE Kiro_Upstream_Provider SHALL throw an error with a descriptive message before attempting any API call.


### Requirement 19: Canonical Response Enforcement

**User Story:** As a developer, I want the Kiro upstream provider to always return parsed canonical types, so that inbound providers receive responses in a format they can process regardless of the passthrough flag.

#### Acceptance Criteria

1. THE Kiro_Upstream_Provider SHALL ignore the `passthrough` flag on Canonical_Request and always parse the Kiro API response into Canonical_Response or Canonical_StreamResponse types.
2. THE Kiro_Upstream_Provider SHALL never return a Canonical_PassthroughResponse, because the Kiro wire format (AWS event-stream with Kiro-specific JSON events) is not compatible with any inbound provider's native format (neither Claude Messages API nor OpenAI Responses/Chat Completions API).
3. WHEN an inbound provider sets `passthrough: true` on the Canonical_Request, THE Kiro_Upstream_Provider SHALL still perform full canonical parsing and return a Canonical_Response (for non-streaming) or Canonical_StreamResponse (for streaming).
4. THE Kiro_Upstream_Provider SHALL only be used with the Claude_Inbound_Provider in phase 1. THE OpenAI_Inbound_Provider is not supported with Kiro upstream because it only handles Canonical_PassthroughResponse and Canonical_ErrorResponse, returning HTTP 500 for any other result type. THE Bootstrap SHALL enforce this by not registering the OpenAI_Inbound_Provider when `UPSTREAM_PROVIDER` is `kiro` (see R14 criterion 6).

### Requirement 20: Bracket-Style Tool Call Parsing

**User Story:** As a developer, I want the provider to detect tool calls embedded in text content, so that models that emit tool calls in bracket format are handled correctly.

#### Acceptance Criteria

1. WHEN processing a non-streaming (collected) response and the accumulated response text contains patterns matching `[Called {function_name} with args: {json}]`, THE Kiro_Stream_Parser SHALL extract these as tool calls.
2. THE Kiro_Stream_Parser SHALL parse the JSON arguments within bracket-style tool calls, handling nested braces and quoted strings correctly.
3. THE Kiro_Stream_Parser SHALL deduplicate tool calls that appear in both structured events and bracket-style text, preferring the version with non-empty arguments.
4. THE Kiro_Stream_Parser SHALL only extract bracket-style tool calls when the parsed function name exists in the `effectiveTools` set provided to the parser. WHEN a bracket-style tool call references a function name not in `effectiveTools`, THE Kiro_Stream_Parser SHALL preserve it as text content rather than extracting it as a tool call. This prevents emitting `tool_use` blocks for tools the client did not provide, which can occur in no-tools/toolChoice-none requests or when Kiro emits a bracket call for an unknown function name.
5. THE Kiro_Stream_Parser SHALL apply bracket-style tool call extraction only to non-streaming (collected) responses. For streaming responses, bracket-style patterns in `text_delta` events SHALL be preserved as text content because they have already been yielded to the client. Bracket extraction in streaming mode would require buffering all text until stream end, which conflicts with incremental streaming. This is a known limitation — in streaming mode, bracket-style tool calls appear as visible text rather than structured `tool_use` blocks. In non-streaming mode, the full response text is available before the `Canonical_Response` is constructed, so bracket extraction can safely identify and convert bracket patterns. Matched bracket-style tool calls SHALL be removed from the accumulated text content and represented as `Canonical_ToolCallBlock` entries in `Canonical_Response.content` with `type: "tool_call"`, `id` generated as `fc_{uuid}`, `callId` generated as `toolu_{uuid}`, `name`, and `arguments`. The `id` is the item-level identifier (analogous to the Codex function_call item id), while `callId` is the tool_use id that Claude Code uses for tool_result matching. The remaining text (with bracket patterns removed) becomes a `Canonical_TextBlock` in `Canonical_Response.content`. The extracted `Canonical_ToolCallBlock` and `Canonical_TextBlock` entries SHALL be ordered in `Canonical_Response.content` according to their original position in the accumulated text. For text containing multiple bracket patterns (e.g., `text1 [Called a...] text2 [Called b...] text3`), the content array SHALL be: `[TextBlock("text1"), ToolCallBlock(a), TextBlock("text2"), ToolCallBlock(b), TextBlock("text3")]`. Empty text spans between or around bracket patterns SHALL be omitted. This is consistent with R9's contract that non-streaming returns a `Canonical_Response` with `content: Canonical_ContentBlock[]`, not events.
6. THE Kiro_Stream_Parser SHALL only extract a bracket-style tool call when BOTH the function name exists in `effectiveTools` (per criterion 4) AND the JSON arguments parse successfully. WHEN a text span matches the bracket pattern format (`[Called {function_name} with args: {json}]`) but the JSON arguments are malformed, truncated, or fail to parse, THE Kiro_Stream_Parser SHALL preserve the entire span as text content rather than extracting it as a tool call. This prevents content loss from partial pattern matches in normal prose or from truncated model output where the bracket pattern started but the JSON was not completed. The extraction logic SHALL confirm JSON parse success BEFORE removing the matched span from the accumulated text, ensuring that a failed parse never discards content.

### Requirement 21: Claude Messages Compatibility Contract

**User Story:** As a developer, I want the Kiro upstream canonical events to produce valid Claude SSE streams when processed by the existing Claude inbound provider, so that Claude Code receives correctly formatted responses end-to-end.

#### Acceptance Criteria

1. THE Kiro_Stream_Parser canonical events MUST produce a valid Claude SSE event sequence when processed by the existing `claudeCanonicalStreamResponse` function in the Claude_Inbound_Provider: `message_start` → (`content_block_start` → deltas → `content_block_stop`)* → `message_delta` → `message_stop`.
2. WHEN the Kiro response contains thinking content, THE Kiro_Stream_Parser SHALL yield canonical events that produce the Claude thinking sequence: `content_block_start(thinking)` → `thinking_delta` events → `thinking_signature` → `content_block_stop`, followed by text content blocks.
3. WHEN the Kiro response contains structured tool call events (content/toolUse events from the Kiro API, not bracket-style text patterns), THE Kiro_Stream_Parser SHALL yield canonical events that produce the Claude tool_use sequence: `content_block_start(tool_use)` → `tool_call_done` (which the Claude inbound converts to `input_json_delta`) → `content_block_stop`, with `stop_reason` set to `tool_use`. Note: Bracket-style tool calls in streaming mode are preserved as text per R20 criterion 5 and do not produce `tool_call_done` events.
4. THE Kiro_Stream_Parser SHALL yield a `usage` Canonical_Event with `outputTokens`. The `inputTokens` value SHALL also be included in the canonical `usage` event when available. However, in streaming responses, the existing `claudeCanonicalStreamResponse` function hardcodes `input_tokens: 0` in the `message_start` event and only uses `outputTokens` from the canonical `usage` event in the `message_delta` event. The `inputTokens` value will be reflected in non-streaming Claude responses (via `canonicalResponseToClaudeMessage`) but not in streaming responses. This is a known limitation of the existing Claude inbound streaming formatter. Phase 1 SHALL track this as a follow-up improvement. A future phase SHOULD update `claudeCanonicalStreamResponse` to accept an optional `inputTokens` value from the Canonical_StreamResponse metadata or from the first `usage` canonical event, and use it in the `message_start` event instead of hardcoding 0.
5. THE Kiro_Stream_Parser MAY yield `content_block_start` and `content_block_stop` events with sequential block indices for future use, but the existing `claudeCanonicalStreamResponse` does not consume these events. The `claudeCanonicalStreamResponse` manages its own `contentIndex` internally. The canonical events that `claudeCanonicalStreamResponse` actually consumes are: `text_delta`, `thinking_delta`, `thinking_signature`, `tool_call_done`, `server_tool_block`, `usage`, `message_stop`, `completion`, and `error`. Block boundary events are optional and do not affect Claude SSE output.
6. FOR ALL valid Kiro streaming responses, the canonical events produced by Kiro_Stream_Parser SHALL be consumable by `claudeCanonicalStreamResponse` without errors, producing a complete Claude SSE stream ending with `message_stop` (compatibility contract property).
7. The Claude Messages compatibility contract (R21) explicitly excludes `toolChoice: required` and named tool `toolChoice` behavior. Kiro does not support forced tool selection, so responses to requests with `toolChoice: required` or named tool may not match Claude's guaranteed tool-use behavior. This is a known phase 1 limitation documented in R6 criterion 18. WHEN a client sends `toolChoice: required`, the Kiro upstream provider includes all tools normally but the model MAY return a text-only `end_turn` response instead of a guaranteed tool call. WHEN a client sends a named tool `toolChoice`, the Kiro upstream provider narrows `effectiveTools` to the named tool but the model MAY still return a text-only response. These deviations from Claude's `toolChoice` contract are acceptable in phase 1 because rejecting `required` would break Claude Code basic functionality (R6 criterion 18).
8. The Claude Messages compatibility contract (R21) explicitly excludes Claude `thinking` configuration (`body.thinking: { type: "enabled", budget_tokens: N }`). The existing Claude inbound provider's `claudeToCanonicalRequest` function does NOT map `body.thinking` into `Canonical_Request`. The `reasoningEffort` field in `Canonical_Request` is only populated from `body.output_config?.effort`, which is a different mechanism. Therefore, requests using `thinking: { type: "enabled", budget_tokens: N }` will NOT trigger thinking tag injection (R6 criterion 22) and will return plain text 200 responses without thinking blocks. This is a known limitation of the existing Claude inbound canonical mapping, tracked for a subsequent phase that would update the Claude inbound provider to map `thinking` configuration into `Canonical_Request`.

### Requirement 22: Provider Switch Runtime Contract

**User Story:** As a developer, I want runtime labels and app state paths to adapt when switching between Codex and Kiro upstream providers, so that the user experience reflects the active provider.

#### Acceptance Criteria

1. THE Bootstrap SHALL pass provider metadata (including a display label such as `"Codex"` or `"Kiro"`) to the runtime via the bootstrap return value. This is a SHOULD (recommended for a future phase) and is not required for phase 1.
2. THE runtime SHOULD use the provider display label in the root JSON response, health endpoint, test-connection endpoint, and console startup output, replacing hardcoded `"Codex"` references. This is recommended for a future phase and is not required for phase 1.
3. THE Bootstrap SHALL return an `authFile` path for Kiro mode that points to a synthetic state file in the app state directory (e.g., `~/.codex2claudecode/kiro-state.json`), not the Kiro credential source path, so that request logs and other app state are written to the correct location.
4. Phase 1 SHALL accept hardcoded `"Codex"` labels in the runtime as a known limitation. THE runtime SHALL continue displaying `"Codex"` labels when `UPSTREAM_PROVIDER` is `kiro`, and this requirement SHALL be tracked for a subsequent phase. This ensures R14 and R15 "no runtime changes" constraints are honored.
5. Phase 1 SHALL accept that the runtime root JSON endpoint list may advertise `/v1/responses` and `/v1/chat/completions` even when they are not registered in Kiro mode (per R14 criterion 6). This is a cosmetic issue — the endpoints correctly return 404 when not registered. This SHALL be addressed when provider metadata is implemented in a future phase.

## Design-Phase Notes

The following items were identified during requirements review and should be addressed during the design and implementation phases. They do not block requirements completion.

### 1. Corrupt companion file fallback

R1 criteria 4 and 7 handle the case where the Device_Registration_File is missing, but do not explicitly address corrupt or partially valid companion files. During design, the Kiro_Auth_Manager should handle invalid JSON, unreadable files (permission errors), and partial fields (e.g., a file containing `clientId` but missing `clientSecret`) in the Device_Registration_File by logging a warning and proceeding without OIDC credentials, consistent with the missing-file fallback behavior in R1 criterion 7. This prevents a corrupt companion file from causing an unhandled exception during initialization.

### 2. Named toolChoice conversion scope

R6 criterion 3, step (3.75) converts non-effective tool content "for named toolChoice" across historical messages. During design, ensure this conversion applies to the ENTIRE pre-split message array — not just the messages that become "history" after the split (step 7). The final tool-result turn that becomes `currentMessage` may also reference non-effective tools (e.g., a tool result for a tool not in the narrowed `effectiveTools`). If step (3.75) only processes history, the `currentMessage` could contain structured `toolResults` for tools not in `effectiveTools`, which may cause Kiro validation failures.

### 3. Claude SSE contract wire names

R21 uses canonical event names (`thinking_signature`, `tool_call_done`) to describe the compatibility contract. However, the actual `/v1/messages` SSE wire format uses different names after the Claude formatter processes canonical events: `thinking_signature` becomes `signature_delta`, `tool_call_done` becomes `input_json_delta`, etc. During design, acceptance test fixtures should assert the actual SSE wire shape produced AFTER the Claude formatter (`claudeCanonicalStreamResponse`), not the canonical event names. R8 and R10 correctly specify canonical event names for the Kiro_Stream_Parser output; R21 fixtures should verify the end-to-end Claude SSE output.

### 4. Structured output (textFormat) requests

The Claude inbound provider maps `output_config.format` from inbound requests into `Canonical_Request.textFormat`, but the Kiro payload conversion pipeline does not currently handle or reject this field. During design, the Kiro_Upstream_Provider should either: (a) return a clear 400 error with a message indicating that structured output / `textFormat` is not supported by the Kiro upstream provider, or (b) document this as a known limitation where `textFormat` is silently ignored and the model returns unstructured text. Option (a) is preferred to avoid silent behavior differences between Codex and Kiro upstream providers.
